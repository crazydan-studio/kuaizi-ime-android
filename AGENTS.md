# Agent 工作准则

本文档定义参与项目开发的 AI Agent 必须遵循的工作准则。

---

## 1. 执行任务前先与远程仓库同步

每次开始执行任务前，必须先将本地工作树与远程仓库同步，确保基于最新代码工作：

```bash
git stash
git pull origin refactor-kotlin
git stash drop  # 仅当 stash 内容仅为文件权限差异时
```

若 stash 包含实质性内容（非仅权限差异），需先评估是否与远程更新冲突，再决定处理方式。

---

## 2. 按变更意图分别提交

所有提交必须按照不同的变更意图分别进行，禁止将多个不相关的变更混合在同一个提交中。

**正确的做法**：

- 重命名类/字段 → 单独提交
- 修正文档内容 → 单独提交
- 调整文件权限 → 单独提交
- 修复链接引用 → 单独提交

**提交信息格式**：遵循项目 Git 规范（`<type>(<scope>): <subject>`），详见 [代码规范 - Git 规范](docs/ai-agent/skills/code-conventions.md)。

---

## 3. 保证普通文件权限始终为 644

所有普通文件（非脚本、非可执行程序）的权限必须保持为 `644`。脚本文件（如 `.sh`）保留可执行权限。

在提交前检查并修正文件权限：

```bash
# 修正普通文件权限（排除 .git 目录和脚本文件）
find . -not -path './.git/*' -type f -perm /111 -not -name '*.sh' -exec chmod 644 {} +
```

---

## 4. 编码和设计前先阅读规范文档

在开始编码或设计工作之前，必须先阅读以下规范文档，确保工作成果符合项目既定约定：

| 文档 | 路径 | 内容 |
|------|------|------|
| AI 技能库索引 | `docs/ai-agent/skills/index.md` | 核心原则、技能文档列表与编写规范 |
| Kotlin 最佳实践 | `docs/ai-agent/skills/kotlin-best-practices.md` | Kotlin 2.3.20 特性使用、惯用写法、避坑指南 |
| Compose 最佳实践 | `docs/ai-agent/skills/compose-best-practices.md` | Compose BOM 2026.04.01 特性使用、IME 场景最佳实践 |
| 代码规范 | `docs/ai-agent/skills/code-conventions.md` | 命名、格式、注释、错误处理、Git 规范 |
| 命名规范 | `docs/ai-agent/v4/design/architecture/020-naming-conventions.md` | 项目级命名约定、前缀规则、已更名对照 |
| 文档组织结构 | `docs/ai-agent/index.md` | 文档类型、目录结构、更新策略 |

---

## 5. 修改设计文档时全面检查关联项

调整、修改、改进设计文档时，必须全面检查所有文档和 PlantUML 图表，确保其中的关联项和引用均被同步修改。

**必须检查的范围**：

1. **同一模块内的其他文档**：确认相关概念、类名、字段名在模块的其他文档中也被同步更新
2. **PlantUML 图表**：确认 `docs/ai-agent/v4/design/diagrams/` 下的所有相关图表中的类名、字段名、关系标注均被同步更新
3. **跨模块文档**：确认上游/下游模块的文档中对被修改概念的引用也被同步更新（如引擎层类名变更需同步到 UI 层和应用层文档）
4. **索引文档**：确认各 `index.md` 中的描述文本已被同步更新
5. **技能文档**：若变更涉及项目命名或设计约定，需同步更新 `docs/ai-agent/skills/` 下的相关文档
6. **迁移映射文档**：确认 `docs/ai-agent/v4/design/migration/` 下的映射表已被同步更新

**操作流程**：

1. 使用搜索工具查找被修改概念在所有文档和图表中的出现位置
2. 逐一确认每处出现是否需要同步修改
3. 执行修改后再次搜索，确认无遗漏
4. 修改完成后，在提交前做最终全局搜索验证

---

## 6. 不确定的修改需用户确认

遇到以下情况时，必须先与用户确认后再执行变更：

- 变更意图存在歧义，存在多种合理的理解方式
- 修改可能影响多处代码或文档，不确定是否需要全部同步
- 涉及架构或设计决策的修改，可能引入破坏性变更
- 对历史变更的上下文不确定，无法判断修改是否与既有设计冲突
- 用户需求中提到的概念在现有文档中找不到对应定义

**确认方式**：向用户说明不确定的具体内容、可能的处理方案及各自的影响，由用户决定最终方案。
